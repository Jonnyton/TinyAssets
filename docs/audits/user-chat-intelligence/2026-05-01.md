### Pattern 1: provider_exhausted class — REFRAMED 2026-05-01

**Original framing was wrong.** The original recommendation was to
populate `GROQ_API_KEY` / `GEMINI_API_KEY` / `XAI_API_KEY`. **That
is wrong-by-default.** Per host pushback + dev-2's post-audit reframe
based on commit `b8d30ac` (2026-04-29 "fix: enforce subscription-only
daemon provider default"):

- `b8d30ac` is **design-aligned, not a regression**. It enforces
  three load-bearing principles: commons-first architecture (cloud
  daemon is subscription-only; users BYOK to their own daemons),
  capacity-bounded host fleets (cloud daemon is the host's always-on
  starter, not THE platform substrate), hostless-BYOK alpha (PLAN.md
  L214 — always ready for next user/daemon fleet without putting
  host on the hook for users' LLM costs).
- Three-layer enforcement: constructor gate (`require_api_key_provider
  _opt_in`) + router filter (`_apply_api_key_provider_policy`) +
  entrypoint env-strip (`subprocess_env_without_api_keys()`).
  `GROQ_API_KEY` etc. are deliberately suppressed even if set.

**The real architectural question (BUG-038 reframe):** was Castles II
*expected* to run on the cloud subscription chain, or was it
user-driven work that should have routed to a user-host or paid-market
bid? **If the dispatcher silently routes user work to the cloud
subscription chain, that's the architectural gap — not a missing
API key.**

Three observed instances of the failure:
- **BUG-038** (Castles II, run `28479d8ddfb44488`, `candidate_discovery`)
- **BUG-039** (Echoes intake, run `dd0e9d48d3204b8c`,
  `make_source_manifest`) — same exhaustion class, different universe
- **BUG-041** (same run as BUG-039, node_statuses contradiction) —
  the secondary state-contract bug

Live daemon status (`mcp_probe.py status` 2026-05-02): 20+ universe
cycles in past 8 hours all completed with `reason=worldbuild_stuck,
words=0, chapters=0`. Daemon polling + accepting requests, but not
producing output. `api_key_providers_enabled: false` confirms the
post-`b8d30ac` policy is live in prod.

**Platform-side fix (architecturally aligned, host-action):** ensure
`WORKFLOW_CODEX_AUTH_JSON_B64` is populated + current in
`/etc/workflow/env` (the legacy `OPENAI_API_KEY` → `codex login`
path is intentionally retired; subscription codex auth flows through
the base64-encoded auth bundle), include the `claude` CLI binary in
the Docker image (BUG-025 fix #1), and bring up Ollama either
in-container or at a network-reachable URL (set `OLLAMA_HOST`).
That gives the chain a working second + third entry beyond codex.

**Path B (allow API keys) is a billing-semantics choice, not an
architecture-aligned fix.** Flipping `WORKFLOW_ALLOW_API_KEY_
PROVIDERS=1` AND populating GROQ/GEMINI/XAI rebuilds the 6-entry
chain but shifts billing from subscription quota to API-key credits.
Only consider this if host explicitly wants this daemon on API-key
billing; otherwise the Path A subscription-chain fix is the
architecturally aligned answer.

## Was BUG-038 user-driven or daemon-driven? (lead's specific Q)

**Both, sequentially — but the architecturally interesting question
is "should it have been on the cloud chain at all?":**

1. The bug was *triggered by a chatbot* (chatbot-filed per
   `reported_by: chatbot` on the BUG-038 page). A real user, via
   their chatbot client, called `extensions.run_branch` with
   Castles II inputs. So initial filing = chatbot acting on a
   user's behalf via live MCP.
2. The bug is *recurring autonomously* now: the cloud daemon's
   continuous universe-cycle loop hits the same provider_exhausted
   class on every cycle (`worldbuild_stuck, words=0, chapters=0`
   × 20+ in past 8 hours). So once filed, it repeats without user
   action.

**Did anyone hit it more than once?** Yes — BUG-039 is the
near-identical failure on a different universe (Echoes), confirmed
same root cause class. And the daemon's autonomous loop hits it
every cycle. The bug is not a one-off; it's the live state of
the production daemon.

**The architectural question (per host pushback 2026-05-01):**
was Castles II *expected* to run on cloud subscription chain at all,
or was it user-driven work that should have routed to a user-host
or paid-market bid? The dispatcher routing logic is the load-bearing
question, not the chain depth. If user work silently lands on the
cloud subscription chain, the architecture is leaking — `b8d30ac`
explicitly says cloud daemon is host-funded starter capacity, not
the place user work runs at scale.
