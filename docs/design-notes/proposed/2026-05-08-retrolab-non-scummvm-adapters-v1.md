# RetroLab Non-ScummVM Runtime Adapters V1

## 1. Recommendation Summary

Adopt the wiki plan as a proposed RetroLab adapter expansion, not as immediate
runtime implementation. The design adds two adapter families to the RetroLab
catalog model:

- `retroarch-portable-v1`, initially covering Atari 2600 through Stella and
  NES/Famicom through Mesen.
- `fs-uae-aros-v1`, covering Amiga PD/freeware through FS-UAE with an open
  AROS Kickstart replacement.

Both adapters should remain community-authored design artifacts until the
runner and linter patches they depend on are available. No new Workflow MCP
primitive is justified by this plan: the platform should keep the generic
fetch, extract, config generation, process launch, log wait, input replay, and
screenshot proof capabilities, while RetroLab supplies the adapter records and
recipes as commons content.

## 2. Classification

Issue #348 was filed as `docs-ops` with mechanical shape. The source page is a
wiki `plan` with `status: design-spec`, so the smallest useful repo change is
this proposed design note. It records the plan in the repo review path without
shipping runtime code, new MCP actions, or platform policy.

This is not a bug. The recent issue comments describe an auto-fix writer
failure: Codex subscription auth was present, but the automated writer did not
open a PR. This branch is the manual Codex bridge for that terminal no-PR
writer-failure class.

## 3. Fit Against PLAN.md

Minimal primitives: pass. The plan does not require family-specific Workflow
tools such as "run Atari 2600" or "run Amiga". It composes from generic runner
capabilities and adapter metadata.

Community-build over platform-build: pass. Runtime adapters, per-game recipes,
proof regexes, and acceptance ladders are commons content that users and
daemons can remix.

Commons-first architecture: pass with one caveat. Public adapter metadata,
hash pins, proof recipes, and legal notes can live in the commons. User-owned
ROMs and proprietary firmware must stay local to the host or be provided only
through rights-cleared deployment paths.

User capability axis: local-app first. These adapters need local execution,
filesystem writes, and emulator windows. Browser-only users may consume the
results through published proofs, but cannot run the local proof harness
without a host.

Engine/domain boundary: RetroLab should remain a domain/catalog layer over the
generic runner. Adapter records should not leak emulator-family concepts into
`workflow/` runtime interfaces beyond generic artifact, process, input, and
proof primitives.

## 4. Adapter Shape

### RetroArch Portable

The `retroarch-portable-v1` adapter should pin:

- RetroArch portable Windows runtime artifact from the official build host.
- Libretro Stella core for Atari 2600.
- Libretro Mesen core for NES/Famicom.
- Deterministic `retroarch.cfg` generated at install time.
- Launch templates that pass `-L`, the selected core DLL, the ROM path, verbose
  logging, and a per-run log file.

The proof contract should require engine initialization, core load, ROM load,
active frame loop evidence, a non-uniform screenshot, clean process shutdown,
and an unbroken audit chain. Version, size, and SHA-256 values remain
`NEEDS_PIN` until a curator pins a concrete adapter version against current
upstream artifacts.

### FS-UAE With AROS Kickstart Replacement

The `fs-uae-aros-v1` adapter should pin:

- FS-UAE Windows runtime artifact.
- A legally clean AROS m68k Kickstart replacement artifact, if a canonical
  distribution URL and hash can be verified.
- `Portable.ini` plus per-game `.fs-uae` config files generated by the runner.
- Launch templates that start FS-UAE with the generated config file.

The proof contract should require FS-UAE initialization, ROM/Kickstart
replacement load, floppy insertion, Amiga environment boot evidence, rendering
loop evidence, a non-uniform screenshot, clean process shutdown, and an
unbroken audit chain.

The adapter must not bundle proprietary Kickstart ROMs. Cloanto/Amiga Forever
or user-owned ROM support can be documented as optional user-local paths, but
the commons adapter must be usable only with legally redistributable artifacts
or explicit rights-cleared deployment files.

## 5. Dependencies And Gates

The wiki plan names these blocker classes. They should be treated as gates
before any implementation PR claims a full pass:

- `RUNNER-006` / PR-019: write deterministic text config files.
- `RUNNER-007` / PR-020: extract 7z archives for RetroArch portable.
- `RUNNER-008` / PR-021: normalize launch log contracts.
- `RUNNER-009` / PR-022: wait for stable windows or frames.
- `RUNNER-010` / PR-023: wait on log markers.
- `RUNNER-011` / PR-024: deterministic input replay.
- PR-026: linter support for `cores` and `firmware` fields.
- PR-027: platform-specific adapter gates.

Partial implementation is acceptable only if it is labeled as a near-pass and
the missing gate is explicit in the recipe output.

## 6. Acceptance Criteria

A later implementation branch may close this design only when it proves:

- Adapter records conform to the universal `RuntimeAdapter` interface.
- All runtime/core/firmware artifacts are version-pinned with SHA-256 hashes.
- Legal evidence is recorded for every redistributable artifact.
- A sample Stella recipe reaches a running frame loop and captures a
  non-uniform screenshot.
- A sample Mesen recipe reaches a running frame loop and captures a
  non-uniform screenshot.
- A sample FS-UAE + AROS recipe reaches an Amiga boot/game-frame proof or is
  explicitly marked near-pass with a concrete missing artifact/gate.
- No proprietary ROM or copyrighted game media is committed.

## 7. Open Questions

- Which exact RetroArch stable release and core snapshot should be pinned for
  the first adapter version?
- What is the canonical, hashable AROS m68k Kickstart replacement distribution
  URL for FS-UAE use?
- Does FS-UAE's current Windows portable package include a usable AROS ROM, or
  must the adapter fetch it separately?
- Which public-domain/freeware Atari 2600, NES, and Amiga samples provide the
  cleanest first proof recipes?
- Should headless `--max-frames` proof be a required RetroArch gate or only an
  optimization once input replay exists?

## 8. Non-Goals

- No runtime code changes in this branch.
- No new user-facing MCP action.
- No hosted proprietary firmware.
- No commercial-game compatibility claim.
- No claim that source-page upstream versions are still latest; all upstream
  versions and hashes require pin-time verification.
