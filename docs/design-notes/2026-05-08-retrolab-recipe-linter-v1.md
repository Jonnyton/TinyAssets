# RetroLab Recipe Linter v1

Community wiki source:
`pages/plans/retrolab-recipe-linter-v1.md`, filed as Issue #350. This
repository note preserves the requested plan for coding sessions without
promoting it to canonical `PLAN.md` truth.

## Classification

Request kind: docs/ops.

Smallest useful repo change: preserve the linter contract and the gate evidence
requirements that a future RetroLab recipe implementation must satisfy. No
runtime code change is implied by this issue.

## Goal

RetroLab recipes should be forkable, lawful, and reproducible instructions for
restoring or running old software. The linter v1 is a pre-publication gate for
those recipes. It prevents recipes from being accepted when they contain piracy
instructions, omit source integrity hashes, prove only that a title screen
appeared, or depend on manual steps that cannot be replayed by another user or
daemon.

## Linter Verdict Shape

The linter should emit the standard gate-branch verdict fields:

```yaml
verdict: pass | send_back | reject | hold
verdict_evidence:
  reason: string
  evidence_refs: [string]
  suggestions: [string]
```

Use `reject` for piracy or bypass instructions. Use `send_back` for missing
hashes, incomplete proof, or vague manual instructions that are fixable by the
recipe author. Use `hold` only when the recipe depends on an external rights or
availability decision that the author cannot resolve in the current pass.

## Required Gates

### Lawful Source Gate

The recipe must not include ROMs, disk images, BIOS files, product keys,
cracks, DRM bypass steps, direct pirate-site links, or instructions that tell a
user how to obtain copyrighted assets unlawfully.

Acceptable source language points users to rights-cleared media, official
stores, publisher releases, user-owned original media, preservation projects
with explicit redistribution rights, or host-local paths supplied by the user.

Required evidence:

- source description for every game binary, firmware, disk image, or data file;
- license or ownership basis when redistribution is claimed;
- no embedded copyrighted binary content in the recipe or generated artifacts.

### Hash Completeness Gate

Every required external binary artifact must have a stable identifier. At
minimum, record SHA-256 for files the recipe consumes. If a preservation
community commonly uses another checksum for the asset, the recipe may include
that checksum too, but SHA-256 remains required.

Required evidence:

- filename or logical artifact name;
- SHA-256 hash;
- artifact role, such as game disk, installer, firmware, patch, data pack, or
  emulator dependency;
- whether the artifact is user-supplied, downloaded from a lawful URL, or
  generated by an earlier recipe step.

### Playability Proof Gate

Title-screen-only proof is insufficient. A recipe passes only when evidence
shows meaningful interaction beyond initial boot.

Required evidence should include at least one of:

- a run trace that reaches controllable gameplay or an application state beyond
  a menu/title screen;
- input evidence, such as keypress/controller events paired with changed game
  state;
- screenshot, video, save-state, or log evidence after the first interactive
  transition;
- emulator or runtime configuration used for the proof.

Recipes for non-game software should prove the equivalent useful state: opening
or booting alone is not enough if the program's primary workflow has not been
exercised.

### Reproducible Steps Gate

Manual steps must be explicit enough for another user or daemon to repeat. A
recipe cannot pass with vague instructions such as "configure the emulator",
"find the right ROM", "click through setup", or "adjust settings until it
works".

Required evidence:

- exact commands, menu choices, configuration keys, or UI steps;
- expected outputs or state transitions after each important step;
- declared host assumptions, such as OS, emulator version, firmware path, and
  controller/input mapping;
- known nondeterministic steps documented as `hold` or `send_back`, not hidden
  inside a passing recipe.

## Minimum Recipe Metadata

A lintable recipe should carry:

```yaml
retrolab_recipe:
  title: string
  target_platform: string
  emulator_or_runtime: string
  host_requirements: [string]
  artifacts:
    - name: string
      role: string
      source_kind: user_supplied | lawful_url | generated
      sha256: string
      rights_note: string
  steps:
    - id: string
      action: string
      expected_result: string
  proof:
    kind: screenshot | video | log | save_state | trace
    beyond_title_screen: true
    evidence_refs: [string]
```

## Implementation Implications

Future RetroLab implementation work should keep this linter narrow and
deterministic:

- Treat piracy and bypass detection as a hard safety gate, not a style warning.
- Prefer structured recipe metadata over prose scraping whenever the recipe
  format can supply it.
- Keep hash validation mechanical: a missing SHA-256 is a linter failure even
  when the prose looks otherwise complete.
- Require proof that demonstrates a useful post-boot state before publishing or
  bounty settlement.
- Emit fix suggestions that tell authors which field or evidence item is
  missing, without suggesting unlawful acquisition paths.

## First Follow-Up Candidates

1. Define the concrete recipe schema file and fixture examples.
2. Add a documentation-only lint checklist to the RetroLab recipe template.
3. Implement a standalone linter over recipe metadata and markdown evidence.
4. Add fixtures for four failure classes: piracy, missing hashes,
   title-screen-only proof, and vague manual steps.
5. Wire the linter as a gate before recipes can be promoted, discovered, or
   used for bounty settlement.

## Open Questions

- Which RetroLab recipe schema owns the metadata: YAML frontmatter, a typed
  sidecar file, or a database-backed Branch record?
- What evidence formats should count as proof for emulator traces that cannot
  safely include copyrighted pixels?
- Should the linter include a curated allowlist of lawful source domains, or
  only reject known-problem patterns and require human review for ambiguous
  sources?
- What is the minimum proof required for software that is interactive but not a
  game?
