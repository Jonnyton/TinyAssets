# USER-PATH — The Walk from First Launch to Hardware (UX North Star)

**Date:** 2026-07-08
**Status:** Binding UX requirements for all transport work. When an implementation choice conflicts with this walk, the walk wins or the conflict is escalated to the founder — do not quietly diverge.
**The design law (binding on all capability work): primitives + commons, never features.** The platform ships irreducible primitives; every capability is a commons design composed from them. The hardcore user recomposes primitives end to end; the vibe coder runs the commons default and gets the full power free. If a proposed implementation is a "feature" that only the platform can modify, it is the wrong shape — decompose it. (Reference implementations of the law: Dataset Forge, Track G §5b; code-CAD design flows, Track I §I5.)

**The one gesture:** every step reuses the same shape — *declare a goal, see a price, set a cap, own what results.* Any screen that breaks this gesture is wrong.

---

## Step 1 — Attach compute (the resource ledger)

User connects, in any combination: subscription (powers the universe intelligence they talk to), local devices (daemon install + self-benchmark + capability registration), API keys, and the market tier (spend cap per period + price caps, defaulting to "never above cloud ceiling"). Fill order is fixed and visible: **own hardware → subscriptions → own keys → market.** Mental model on the tin: *my stuff first, market fills gaps, never past my caps.*

**SEAM-1 (CLOSED — requirement):** No wallet, chain, or token vocabulary appears at ledger setup. Setting caps requires zero funding. The funding moment is **deferred to first market fill**: at the first purchase, an embedded fiat on-ramp (card → platform credit) appears inline, one screen, with the crypto path available behind an "advanced" disclosure for those who want it. A user who never leaves fiat never learns the settlement rail exists. Acceptance test: a new user reaches their first completed market-priced run having typed a card number once and a wallet address never.

## Step 2a — Design the universe itself (LAYER 0 — the harness of the harness)

Before any workflow exists, the user designs the universe as an entity. Five decisions, each with commons-template defaults so the step can be one tap ("start from the Researcher archetype") or a deep customization:

1. **Disposition** — premise, soul, persona (`has_premise`/`has_soul`/`persona.self_model` are already first-class in the platform).
2. **Body plan** — the agent architecture the universe runs as. REQUIREMENT: TinyAssets does not compete with agent runtimes; it attaches to them. A universe can *wear* an external body — OpenClaw, Hermes-class agents, a plain chat loop — via the MCP connector, with the platform providing goals, graphs, memory catalog, and markets to whatever runtime the user already has. An existing OpenClaw user's onboarding is "connect, don't migrate": their agent gains a universe; nothing is thrown away.
3. **Memory architecture** — platform graph/catalog, local-first files, or hybrid; explicit private/commons boundary set here, not buried in settings.
4. **Surfaces** — where the universe lives: app, messaging channels, terminal. Multi-surface from day one; the universe is the continuity, not the app.
5. **Autonomy envelope** — proactivity (heartbeat cadence, quiet hours) and unprompted-spend allowance. The allowance is drawn against Step 1's caps, never beside them: one budget system, two names.

**SEAM-3 (CLOSED — requirement):** Universe designs (archetypes) are commons artifacts — shareable, remixable, attribution-carrying — and an archetype ships *with* its starter workflow designs so Layer 0 and Layer 1 remix as one unit. The blank-universe cold start must never be the default path; the archetype gallery is.

## Step 2b — Design the workflows (LAYER 1)

Plain-language goal → the universe searches the commons, remixes a graph, presents: the plan, what each node runs on (nodes carry their models — the user's correction that shaped the whole market design), and the cost.

**SEAM-2 (CLOSED — requirement):** The price feed is wired into the *design conversation*, not just execution. The universe-intelligence prompt/tooling MUST consult the quote endpoint (`/v1/price`, `/v1/curve`) while planning, so every proposed design carries a cost line and every redesign shows its delta ("switching this node to Qwen-32B on your own PC: −$0.31/week"). A design presented without a cost line is a defect. This is why the quote endpoint is unauthenticated, cached, and fast — planning-time reads dominate execution-time reads.

## Step 3 — First run (spot)

Own hardware serves what it can; the gap fills at spot under the caps; outcome gates claim with evidence. REQUIREMENT: the first-run receipt shows the fill order in plain words ("your PC did X free; the market did Y for $0.40") — this receipt is where the mental model from Step 1 is confirmed or lost.

## Step 4 — Big batch (forwards)

For deferrable work, the universe offers the trade in plain language: "~$31 now with price risk, or $24 locked for Friday night." REQUIREMENT: the words *forward, future, derivative* never appear in the default UX. The concept ships as "cheaper if you can wait."

## Step 5 — Train a model (Track F)

The universe proposes a fine-tune when repeated corrections make one economical; license composition (fail-closed) runs before the quote is even shown; checkpoints settle; a capability mints under the user's name; the harness redesigns onto it and the user's costs visibly FALL. REQUIREMENT: the cost-drop moment is surfaced explicitly ("your model now serves this step on your own hardware: −$0.85/week") — it is the flywheel felt from inside, and the single strongest retention event on the platform.

## Step 6 — Pool (Track H)

Commons goals with gate ladders and open pools; contribute → fractional ownership → revenue drips to the ledger. REQUIREMENT: pool terms state run risk on the tin (funders lose the spent portion of a failed run) — one sentence, before the contribute button, always.

## Step 7 — Flip to selling

One toggle on the ledger: serve own + chosen commons capabilities when idle, floor price set. The ledger becomes two-directional with the same instruments. REQUIREMENT: selling is discovered *after* buying has taught the market's shape — never pushed at onboarding.

## Step 8 — Hardware (Track I)

Physical fabrication as paid requests (geo-ranked, per-unit settled); shuttle pools with the same contribute gesture as Step 6; hardware capabilities mint and re-enter the market as instrument classes. REQUIREMENT: atoms use the identical UX vocabulary as tokens — a print job and an inference job present as the same object with different capability strings.
