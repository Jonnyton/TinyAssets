## WHAT WE FOUND

### Scope of Brain Substrate
- 42 modules across 6 subsystems
- 12,306 lines of production code
- 3,532+ lines of test coverage
- 100% of code is production-wired (no orphaned modules)

### Subsystems

1. **MEMORY** (16 modules, 4,733 lines)
   Three-tier hierarchy: Core → Episodic → Archival
   Central orchestrator: MemoryManager.assemble_context(phase, state)
   Features: 5-tier scoping, sliding-window episodic eviction, fact promotion, reflexion engine

2. **KNOWLEDGE** (7 modules, 1,909 lines)
   Hybrid three-layer retrieval: KG (HippoRAG) + RAPTOR + Leiden
   Schema: entities, edges, facts, communities (all 4-tier scoped)
   Features: entity extraction, Personalized PageRank, community detection, recursive summarization

3. **RETRIEVAL** (5 modules, 1,418 lines)
   Agentic router: query decomposition + multi-backend routing
   Backends: HippoRAG (entity), RAPTOR (thematic), LanceDB (tone/similarity)
   Features: phase-aware gating, fallback chains, LanceDB singleton wrapper

4. **LEARNING** (4 modules, 476 lines)
   Style rule lifecycle: DRAFT → CALIBRATING → ACTIVE → DECAYED
   Outputs: craft cards, criteria discovery, rule promotion/decay
   Features: evidence-based calibration, violation tracking, LLM-based criteria inference

5. **INGESTION** (6 modules, 2,032 lines)
   File routing: ≤5KB to canon/, >5KB to sources/ + synthesis signal
   Pipeline: file detection → extraction → entity extraction → KG integration
   Features: magic byte detection, manifest tracking, image/video/PDF handling

6. **STORAGE** (4 modules, 626 lines)
   Account management, capability limits, file rotation
   Connections: SQLite singleton, LanceDB paths, ephemeral cleanup
   Features: per-account caps, retention tiers (hot/warm/cold)

## CRITICAL ARCHITECTURE PATTERNS

### 1. Memory Manager as Central Orchestrator
Entry point: MemoryManager.assemble_context(phase: str, state: dict) → ContextBundle
Phases: ORIENT, PLAN, DRAFT, EVALUATE (matching graph-core flow)
Budget: 15K tokens (with automatic trimming)
Coordinates: core memory, episodic memory, archival (KG), promotion gates, reflexion engine

### 2. 5-Tier Orthogonal Scoping (Stage 2b, Stage 2c pending)
All tables have (universe_id, goal_id, branch_id, user_id) columns:
  - episodic_facts, scene_summaries (memory/episodic.py)
  - entities, edges, facts, communities (knowledge/knowledge_graph.py)

Stage 2b (current): Columns appended, no predicate filtering
Stage 2c (pending): WORKFLOW_TIERED_SCOPE flag enables read-side filtering

Design document: docs/design-notes/2026-04-15-memory-scope-tiered.md

### 3. Singleton Patterns with Cross-Universe Guards
get_db(path: str) — LanceDB connection
  - Thread-safe via _lock
  - Explicit path required (prevents cwd-relative collision)
  - reset_db() for testing only

KnowledgeGraph(db_path: str) — SQLite connection
  - Same pattern as get_db()
  - Explicit path guard prevents two universes sharing relative paths

### 4. Graceful Fallbacks
ArchivalMemory._try_load_kg(), _try_load_asp()
  - Returns empty results when backends unavailable
  - Allows memory system to bootstrap standalone
  - No hard dependency on KG or Clingo (optional ASP)

### 5. Phase-Aware Context Assembly
MemoryManager._assemble_orient(), _assemble_plan(), _assemble_draft(), _assemble_evaluate()
  - Phase-specific sub-builders
  - Budget tracking: if >15K tokens, trim_to_budget()
  - Debug: WORKFLOW_DEBUG_CONTEXT=1 for detailed logging

## INTEGRATION POINTS

### Within Brain Subsystems
Memory ← → Knowledge:
  - ArchivalMemory delegates hipporag_query, raptor_query, get_open_promises to KnowledgeGraph
  - PromotionGates detects persistent violations → ASP rule candidates

Memory ← → Retrieval:
  - Router consumes MemoryScope for scoped queries
  - VectorStore results feed reflexion_engine for fact evaluation

Memory ← → Learning:
  - LearningSystem integrated into manager.py flow
  - PromotionGates examines style_rules for violation trends

Memory ← → Ingestion:
  - ProgressiveIngestor coordinates canon file processing
  - Extracted summaries stored as episodic memories

Retrieval ← → Knowledge:
  - Router.retrieve() dispatches to HippoRAG, RAPTOR, VectorStore

Knowledge ← → Ingestion:
  - Indexer.ingest_and_index() calls entity_extraction
  - Extracted entities inserted into KnowledgeGraph

### To External Systems
- workflow/api/runs.py: retrieval.router.tiered_scope_enabled
- workflow/api/runtime_ops.py: memory.project.* (get/set/list)
- workflow/api/universe.py: ingestion.core.ingest_file
- workflow/daemon_server.py: storage re-exports
- Graph-core nodes: via state passing + MemoryManager calls

## CONSOLIDATION OPPORTUNITY ANALYSIS

### Current State
6 independent subsystems with tight coupling:
- Memory module imports from Knowledge, Learning, Ingestion, Storage
- Knowledge module imports from Ingestion, Storage
- Retrieval module imports from Memory, Knowledge, Storage
- etc.

### Proposed: Unified workflow/brain/ with Three Implementations
- DaemonBrain: background async processes (state replication, cache warming)
- ChatbotBrain: interactive LLM queries (retrieval-focused, minimal cache)
- CollectiveBrain: shared inference layer (cross-universe lookups, federation)

### High-Value Consolidation Targets
1. MemoryManager + KnowledgeGraph + Router: common orchestration layer
2. Scoping primitives: eliminate (universe_id, goal_id, branch_id, user_id) duplication
3. Models: FactWithContext + enums already shared, centralize
4. Vector store + graph store: both need explicit path guards + scope tiers
5. Learning system: integrates all subsystems, should be core

### Breaking Apart Risks
- Scoping threading: must maintain schema consistency across 4 tables
- Singleton patterns: get_db() and KnowledgeGraph._connect() both require path guards
- Test isolation: in-memory SQLite can't be shared; per-test setup mandatory
- Phase context gating: router's should_use_backend() references phase enums (coupling point)

### Low-Hanging Fruit for Separation
- Image/video extraction: optional plugin (ingestion/image_extractor.py, ingestion/video_extractor.py)
- ASP constraint layer: already optional fallback
- Tools.py: should live in separate MCP service layer (not core brain)

## NEXT STEPS FOR CONSOLIDATION

1. **Map internal dependencies more granularly:**
   - Which classes in Memory actually use which classes in Knowledge?
   - Can we decouple via interface boundaries?

2. **Design the three-implementation split:**
   - DaemonBrain interface + methods
   - ChatbotBrain interface + methods
   - CollectiveBrain interface + methods
   - Shared core module (IFacts, IGraph, ILearning, etc.)

3. **Plan database schema consolidation:**
   - All 4 tables (episodic_facts, entities, edges, facts) have scope columns
   - Can they share a common scope-prefixed table naming scheme?
   - How to migrate without breaking existing universes?

4. **Identify test isolation strategy:**
   - In-memory SQLite is perfect for unit tests
   - Integration tests need fixture setup per test (not shared)
   - Can we share universe fixture builders?

5. **Handle singleton patterns:**
   - get_db() and KnowledgeGraph._connect() both have _lock + _path guards
   - New brain module needs single entry point for both
   - Design: BrainFactory.get_universe_brain(universe_id, db_path) → DaemonBrain | ChatbotBrain | CollectiveBrain?

6. **Consolidation order:**
   - Start with models: FactWithContext + enums (already cross-subsystem)
   - Then scoping: eliminate 4-tier column duplication
   - Then orchestration: MemoryManager + Router integration
   - Then storage: unified path/connection handling
   - Finally: extract domain-specific learnings (learning system goes last)

### Key realization for consolidation
The brain is already well-architected for the three-implementation split:
- Memory (state management) — clean interfaces
- Knowledge (retrieval) — clean interfaces
- Learning (feedback loops) — clean interfaces
- Scoping (authorization) — orthogonal tiers
- Ingestion (ingest pipeline) — clean separation

The consolidation work is primarily:
1. Extracting shared abstractions (IMemory, IGraph, ILearning)
2. Eliminating schema duplication (scope columns)
3. Centralizing singleton management
4. Reordering imports (dependency inversion)
5. Splitting test setup (per-universe fixtures)

No major architectural changes needed. The pieces are already there.
