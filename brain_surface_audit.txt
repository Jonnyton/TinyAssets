## SECTION 7: CROSS-FILE DRIFT & NAMING INCONSISTENCIES

1. **Memory Surface Fragmentation**
   - project_memory_* in extensions.py (runtime_ops.py backend)
   - Subscribe/messaging in extensions.py (runtime_ops.py backend)
   - Universe-level state in universe.py (requests, canon, activity)
   - Wiki in wiki.py (global knowledge)
   - → No unified "remembering" surface; chatbots route differently per domain

2. **Brain Operation Type Naming**
   - wiki.py uses: ingest, query, digest, lint, export (file_bug → GitHub)
   - runtime_ops.py uses: project_memory_*, messaging_*, schedule_*
   - universe.py uses: submit_request, add_canon, give_direction
   - → Inconsistent verb naming; no unified ingest/query/lint/export ontology

3. **State Layers (4 different persistence models)**
   - wiki/ — markdown files in categories, frontmatter metadata, global
   - project_memory/ — JSON per-project, per-universe, versioned
   - messages/ — per-node queues, transient
   - requests/ — per-universe requests, append-only log
   - → Chatbots must understand which surface to use for each intent

4. **Scope Confusion**
   - Wiki: global, shared
   - Project Memory: per-universe, per-project, isolated
   - Messaging: per-node, transient
   - Requests: per-universe, append-only
   - → No single "tell the brain to remember X" action

5. **Missing Collective Operations**
   - search across all universes' project_memory? NO
   - list all subscriptions across universes? NO
   - "who else is working on Y?" → only wiki.search, goals.search
   - → Collective memory limited to wiki; project-level memory is isolated

6. **Messaging vs. Collective Brain**
   - messaging_* is for teammate coordination (internal)
   - wiki is for reference (external)
   - No middle ground for "broadcast to all branches" or "poll team state"
   - → Messaging is not a knowledge surface

7. **Action Name Collisions / Drift**
   - "list" appears in: wiki (list pages), extensions (list nodes), universe (list universes), runtime_ops (list schedules)
   - Each takes different parameters; no unified list() concept
   - → Verb overloading across MCP surfaces

## CONSOLIDATION PRIORITIES

Based on this audit, here are the high-priority consolidation targets:

1. **Unified "remember" surface**
   - Wiki (prose) + project_memory (JSON) need a unified read/write API
   - Current split confuses chatbots; should have single "add_to_brain(scope, key, value)" and "query_brain(scope, query)"
   - Scope ∈ {global, per-universe, per-project}

2. **Brain operation ontology**
   - Standardize ingest/query/lint/digest/export across all surfaces
   - Current names (search, write, consolidate, file_bug, messaging_send) don't form a pattern

3. **Collective vs. Project Memory**
   - Decide: is per-project memory part of collective brain or daemon-local?
   - If collective: surfaced in wiki or separate shared_memory_* actions
   - If daemon-local: remove from collective brain scope; document isolation

4. **Messaging as Knowledge Surface?**
   - messaging_* is currently teammate routing only
   - Could extend to chatbot-facing "team context" or "live sync"
   - Or remove from brain scope; keep as implementation detail

5. **Query World**
   - universe.py:query_world is stub; decide: search universe state, search wiki, or search project_memory?
   - Currently ambiguous

6. **Duplicate "list" Actions**
   - Consolidate list variants (wiki.list, extensions.list, universe.list, runtime_ops.list_schedules)
   - Single unified list(surface, category, query, limit) → cleaner chatbot routing

7. **Ledger Consolidation**
   - Currently only universe.py has ledger; should wiki/project_memory also be ledgered?
   - For auditability, yes; current split = audit gap
