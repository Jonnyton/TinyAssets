## SUBSYSTEM: MEMORY (16 modules, 4,733 lines)

### ARCHITECTURE
- **Tier 1: CoreMemory** (103 lines) — in-memory active context ~8-15K tokens
  Purpose: Phase-specific ephemeral state (characters, world state, promises, rules)
  Access: dict-based with (category, key) tuples
  Lifecycle: Assembled fresh for each node
  
- **Tier 2: EpisodicMemory** (474 lines) — recent hot data, SQLite-backed
  Purpose: Last N chapters' scene summaries, extracted facts, style observations
  Schema: scene_summaries, episodic_facts, style_observations, reflections (all 4-tier scoped)
  Window: Sliding window for eviction, older data → Archival
  Key class: EpisodicMemory (db_path, universe_id, window_chapters)

- **Tier 3: ArchivalMemory** (156 lines) — bridge to KG + ASP
  Purpose: Long-term storage via knowledge graph + constraint solver
  Delegates: hipporag_query(), raptor_query(), get_open_promises(), validate_facts()
  Fallback: Returns empty results when KG/ASP not available
  
- **Manager: MemoryManager** (548 lines) — central orchestrator
  Key method: assemble_context(phase, state) → ContextBundle
  Phases: ORIENT, PLAN, DRAFT, EVALUATE (matching SceneState flow)
  Budget: MAX_CONTEXT_TOKENS = 15,000
  Sub-coordinators: promotion_gates, reflexion_engine
  Key methods:
    - assemble_context() — main entry point
    - store_scene_result() — persists after evaluation
    - run_promotion_gates() — lifecycle transitions
    - run_reflexion() — self-critique on revert
    - evict_old_data() — episodic window maintenance

**Consolidation (346 lines)** — deduplication + merging

**Promotion (124 lines)** — fact/rule/style lifecycle

## SUBSYSTEM: KNOWLEDGE (7 modules, 1,909 lines)

### ARCHITECTURE
Hybrid three-layer retrieval backbone:
1. Knowledge Graph (igraph + SQLite) with HippoRAG Personalized PageRank
2. Leiden community detection for character/faction/plot threads
3. RAPTOR tree synthesis for global/thematic queries

### CORE MODULES

**Knowledge Graph (747 lines)** — entity-relationship store
  Schema:
    - entities: id, name, entity_type, description, created_at (+ 4-tier scope cols)
    - edges: source_id, target_id, relation_type, weight, created_at
    - facts: entity_id, fact_id, text, source_type, ... (+ 4-tier scope cols)
    - communities: community_id, entities (JSON), summary, created_at
  
  Key methods:
    - add_entity(), add_edge(), add_fact()
    - build_igraph() — constructs in-memory igraph for analysis
    - hipporag_query(entities, k) — delegates to hipporag.py
    - raptor_query(query, k) — delegates to raptor.py
    - get_open_promises(overdue) — promise tracking
  
  Scoping: Stage 2b appends (universe_id, goal_id, branch_id, user_id) to all tables
  Thread-safe: _lock guards singleton connection
  Cross-universe guard: explicit path required (prevents cwd-relative collisions)

**HippoRAG (214 lines)** — Personalized PageRank on entity graph
  Algorithm: PPR from seed entities (character/location names) → top-k ranked facts
  Damping: 0.85 (standard)
  Output: [{"entity_id", "ppr_score", "rank"}]
  Use case: "Who does Ashwater know?" "What's the faction's core conflict?"

**RAPTOR (361 lines)** — Recursive abstractive summarization tree
  Structure: RaptorNode (leaf=chunk, level 0; internal=summary, level>0)
  Build: chunks → clustering → cluster summaries → meta-summaries
  Clustering: k-means on embeddings (tunable n_clusters)
  Async: build_raptor_tree(), query_raptor_tree()
  Output: top-level summaries for global/thematic queries
  Provides 72%+ accuracy on multi-hop reasoning (per docstring)

**Entity Extraction (226 lines)** — LLM-based NER + relationships
  Prompt: FICTION_EXTRACTION_PROMPT (detailed in docstring)
  Extracts:
    - Entities: characters, locations, factions, artifacts, magic systems, events
    - Relationships: conflict, alliance, mentorship, parentage, knowledge, etc.
    - Facts: text, source_type, language_type, narrative_function, importance, confidence
  
  Access Tier Enum:
    0 = common knowledge
    1 = insider knowledge
    2 = secret/restricted
    3 = cosmic/metaphysical
  
  Output: {"entities": [...], "relationships": [...], "facts": [...]}
  AliasRegistry class: entity deduplication (pronouns, titles, nicknames)

**Leiden (126 lines)** — Community detection
  Algorithm: igraph.community_leiden() with modularity objective
  Tuning: resolution (denser → higher) + n_iterations (default 10)
  Output: Community objects with entity names + summaries
  Use: Character groups, plot threads, faction networks, world regions
  Async: summarize_communities() — LLM-based description generation

**Models (228 lines)** — shared data types
  Key classes:
    - FactWithContext: fiction-aware fact with truth-value metadata
      Fields: fact_id, text, source_type, narrator, language_type, narrative_function,
              importance, hardness, horizon, provenance, confidence, access_tier
      Methods: is_accessible_to(character_id, knowledge_level), is_valid_at_chapter()
    - Enums: SourceType, LanguageType, NarrativeFunction, TruthValue, FactHardness, FactHorizon
    - GraphEntity, GraphEdge (TypedDicts for igraph integration)
    - Community, SubQuery, RetrievalResult, QueryType (for router)

**Knowledge __init__.py** — empty (7 lines)
  Note: No public exports; modules used directly elsewhere

### KNOWLEDGE TEST COVERAGE
- test_knowledge_graph.py (848 lines): Entity/edge/fact CRUD, igraph construction, HippoRAG/RAPTOR queries, community detection

### EXTERNAL DEPENDENCIES
- igraph (python-igraph): graph algorithms, community detection
- lancedb: vector store for embeddings
- numpy: clustering, embeddings
- yaml: config (limited usage)

## SUBSYSTEM: RETRIEVAL (5 modules, 1,418 lines)

### ARCHITECTURE
Agentic routing layer that decomposes compound queries and routes to appropriate backends.

### CORE MODULES

**Router (628 lines)** — query decomposition + backend routing
  Key functions:
    - decompose_query(query, phase) → list[SubQuery]
      LLM decides query type: entity_relationship | thematic_global | tone_similarity
    - retrieve(query, phase, scope) → RetrievalResult
      Dispatches sub-queries to HippoRAG, RAPTOR, or LanceDB vectors
  
  Decomposition Prompt: DECOMPOSITION_SYSTEM + DECOMPOSITION_PROMPT
  Router class: encapsulates state + provider_call
  _parse_decomposition(): LLM JSON → SubQuery objects
  _simple_decompose(): fallback when LLM decomposition fails
  
  Scope awareness: memory.scoping.MemoryScope threading (Stage 2c only)
  Feature flag: WORKFLOW_TIERED_SCOPE (off in Stage 2b)
  Backend eligibility check: should_use_backend(phase, query_type)

**Vector Store (239 lines)** — LanceDB singleton wrapper
  Singleton pattern: get_db(path) — thread-safe via _lock
  Key classes:
    - VectorStore: table wrapper
      Methods: add(), search(embedding, k), delete(), clear()
      Pre-computed embeddings: externally embedded (LanceDB doesn't call model)
  
  Schema: Vector table with metadata columns (source, chapter, etc.)
  Cross-universe guard: explicit path required (prevents cwd-relative collision)
  reset_db(): test-only reset

**Agentic Search (468 lines)** — hierarchical search + fallback chains
  Key functions:
    - search_with_fallback(query, backends, scope) → list[FactWithContext]
    - hierarchical_search() — retry with simplified query on failure
  
  Backends: HippoRAG → RAPTOR → vectors
  Graceful degradation: returns partial results when backends unavailable
  Scope integration: MemoryScope passed through to backend queries

**Phase Context (76 lines)** — phase-aware backend selection
  should_use_backend(phase, query_type) → bool
  Logic: some backends unavailable in certain phases (e.g., RAPTOR in PLAN)
  Per-phase eligibility: ORIENT, PLAN, DRAFT, EVALUATE

## SUBSYSTEM: LEARNING (4 modules, 476 lines)

### ARCHITECTURE
Observation → promotion cycle for style rules, craft cards, and discovery-based criteria.

### CORE MODULES

**Style Rules (221 lines)** — rule lifecycle + calibration
  Key classes:
    - StyleRule: dimension, rule_text, calibration_score, promotion_state
    - StyleRuleState enum: DRAFT, CALIBRATING, ACTIVE, DECAYED
    - LearningSystem: central rule manager
      Methods: add_rule(), promote_rule(), decay_rule(), get_active_rules()
  
  Lifecycle: DRAFT → CALIBRATING (gather evidence) → ACTIVE → DECAYED (violations)
  Evidence: scene references, acceptance rate, user feedback
  Calibration: automatic promotion after threshold hits (e.g., 5 successful scenes)
  Decay: persistent violations (3+ scenes) → ASP rule candidate (in memory/promotion.py)

**Craft Cards (110 lines)** — compressed lessons from chapter evaluations
  Key classes:
    - CraftCard: card_id, dimension, issue, evidence, recommendation, chapter_number, severity
  
  generate_craft_cards(chapter_state, style_rules) → list[CraftCard]
  Sources: style_observations, quality_trend, rule promotions/decays
  Severity: info | warning | critical
  Purpose: actionable writer feedback (1-5 cards per chapter)

**Criteria Discovery (117 lines)** — automated criterion inference
  discover_criteria(prose, reference_rules) → list[DiscoveredCriterion]
  Infers writing dimensions from prose + reference standards
  Output: DiscoveredCriterion (dimension, importance_estimate, found_in_prose)

**Learning __init__.py** (28 lines) — public API

### INTERNAL CALL GRAPH

Memory ← → Knowledge:
  - ArchivalMemory delegates to KnowledgeGraph (hipporag_query, raptor_query, get_open_promises)
  - KnowledgeGraph.validate_facts() calls constraints.asp_engine (graceful fallback)
  - memory/promotion.py detects ASP rule candidates

  - Router.decompose_query() consumes memory.scoping.MemoryScope
  - RetrievalRouter returns facts to MemoryManager for promotion/reflexion
  - VectorStore queries feed into agentic_search fallback chains

  - LearningSystem integrated into memory/manager.py flow
  - PromotionGates examines style_rules for violation trends

Memory ← → Ingestion:
  - Ingestion output → stored as episodic memory summaries

## CONSOLIDATION OPPORTUNITY ANALYSIS

Current state: 6 subsystems (memory, retrieval, knowledge, learning, ingestion, storage)
Proposal: unified workflow/brain/ with three implementations (DaemonBrain, ChatbotBrain, CollectiveBrain)

High-value consolidation targets:
  1. MemoryManager + KnowledgeGraph + Router: common orchestration layer
  2. Scoping primitives: currently duplicated across memory/retrieval/knowledge
  3. Models: FactWithContext + enums already cross-subsystem shared
  4. Vector store + graph store: both need universe_id + scope tiers
  5. Learning system: integrates with all subsystems, should be core

Breaking apart risks:
  - Scoping threading: all tables need consistent (universe_id, goal_id, branch_id, user_id) columns
  - Singleton patterns: get_db() and KnowledgeGraph._connect() both require explicit path guards
  - Test isolation: in-memory SQLite can't be shared; per-test setup mandatory
  - Phase context gating: router's should_use_backend() references phase enums

Low-hanging fruit for separation:
  - Image/video extraction: can become optional plugin
  - ASP constraint layer: already optional fallback
  - Tools.py: should live in a separate MCP service layer
